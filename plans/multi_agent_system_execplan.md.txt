# ExecPlan: Build a 7-agent research/application system with OpenAI models

## Goal
Build a manager-led multi-agent system for research planning, literature mapping, project design, supervisor targeting, outreach drafting, and internal review.

## Why
The user wants a practical AI system that coordinates up to 7 specialized agents with strong custom prompts, shared state, and reliable orchestration. The system should be useful for PhD/research workflows and later extensible to other complex planning tasks.

## Scope
Implement:
- 1 manager agent:
  - ChiefOfStaff
- 6 specialist agents:
  - TopicStrategist
  - LiteratureCartographer
  - ProjectArchitect
  - SupervisorMapper
  - NarrativeWriter
  - SkepticalReviewer

Use OpenAI models and a modular architecture so prompts, schemas, tools, and orchestration are separated cleanly.

## Non-goals
- Do not build a fully autonomous unsupervised internet crawler.
- Do not implement every future workflow now.
- Do not over-optimize UI before the core orchestration works.

## Architecture
- `src/agents/`
  - one file per agent
- `src/prompts/`
  - one prompt file per agent
- `src/schemas/`
  - pydantic/json schemas for each agent output
- `src/orchestrator/`
  - manager routing logic
  - shared state
  - execution graph
- `src/tools/`
  - web search wrapper
  - file/document wrapper
- `tests/`
  - orchestration tests
  - schema validation tests
  - smoke tests for each agent

## Required behavior
- ChiefOfStaff receives the user goal and decides which agents to call.
- Each specialist returns structured JSON.
- SkepticalReviewer reviews important outputs before they are accepted.
- Shared state persists:
  - goal_profile
  - topic_pool
  - reading_board
  - project_board
  - target_supervisors
  - drafts
  - review_log
  - timeline

## Milestones

### Milestone 2: Core agents

### Milestone 3: Research agents
- Implement LiteratureCartographer and ProjectArchitect.
- Add benchmark map + reading ladder support.

### Milestone 4: Targeting and writing
- Implement SupervisorMapper and NarrativeWriter.
- Add output formats for ranked supervisors and outreach drafts.

### Milestone 5: Full orchestration
- Add review gates.
- Add state updates after each completed stage.
- Add logging/tracing hooks.

### Milestone 6: Validation
- Add tests for:
  - routing decisions
  - schema compliance
  - state transitions
  - review rejection/revision loops

## Constraints
- Keep prompts externalized in files.
- Keep outputs machine-readable.
- Keep agent responsibilities narrow and explicit.
- Prefer a manager-led design over free-form swarm behavior.
- Make it easy to swap models later.

## Done when
- A user can submit a high-level goal.
- The system routes the task through the right agents.
- The system returns:
  - a narrowed topic
  - a literature map
  - a first project concept
  - supervisor targets
  - draft outreach text
  - reviewer critique
- Tests pass for the main orchestration path.

## Decision log
- Manager-led orchestration chosen over peer swarm for control and auditability.
- Prompts stored as files for easier iteration.
- JSON outputs required to support chaining and validation.

## Open questions
- Whether to use Responses API only or Agents SDK + Responses API.
- Whether tool access is global or per-agent.

### Milestone 6 audit findings and fixes
- Issue: deterministic agents could return non-dict payloads and fail with less actionable errors deeper in schema code.
  - Fix: added explicit payload type assertion in `DeterministicAgent.run` raising `SchemaValidationError` with agent context.
- Issue: revision/rejection gating behavior had good `REJECT` coverage but weaker explicit `REVISE` path coverage.
  - Fix: added orchestration test asserting `REVISE` blocks all subsequent delegations.
- Issue: prompt loader behavior was not explicitly tested for custom prompt directories and missing files.
  - Fix: added prompt loader tests for both paths.
- Issue: malformed/partial outputs were not comprehensively covered beyond schema unit tests.
  - Fix: added dedicated hardening tests for malformed and partial deterministic-agent payloads.

### Remaining technical debt
- Orchestrator currently blocks on non-PASS review verdicts without an automatic retry/revision execution loop.
- State transition logic still uses per-agent inline handlers in one method; maintainability could improve with small stage objects if complexity grows.
- Schema layer relies on custom dataclass validation helpers instead of a fully declarative typed schema framework; this is adequate now but may need stronger static typing as agent outputs expand.

### Recommended next steps
1. Add an optional bounded retry loop for `REVISE` verdicts that re-runs only the failed stage with reviewer instructions appended.
2. Introduce typed stage descriptors (agent name, dependencies, state sink key) to reduce duplicated orchestration branching as new agents are added.
3. Add fault-injection integration tests for mid-pipeline exceptions and ensure timeline snapshots remain consistent under failures.

### Follow-up adjustments after hostile-review pass
- Conducted a hostile senior-review style failure analysis against messy real-world inputs in orchestration.
- Concrete failure scenarios identified:
  1. Unbounded delegations from `ChiefOfStaff` can inflate timeline/state and cause noisy, costly runs.
  2. `ChiefOfStaff.state_update` accepted arbitrary list payloads, allowing non-dict timeline entries that can break event consumers expecting dict-shaped records.
  3. Mixed-quality manager state patches (partially valid / partially malformed) were not explicitly audited, reducing diagnosability.
- Implemented hardening patches for the highest-risk scenarios:
  - Added a delegation safety cap (`MAX_DELEGATIONS_PER_RUN = 12`) in orchestration with explicit `delegations_truncated` timeline/trace event.
  - Added strict `state_update` coercion for list-backed state fields to require `list[dict]`; malformed entries are rejected with `chief_state_update_rejected` events instead of being silently applied.
- Added regression tests to lock behavior:
  - delegation truncation and dropped-count accounting
  - rejection logging for non-dict `state_update` list payloads
- Re-ran targeted orchestration tests and full test suite after the hardening updates.
- Blockers: none.
